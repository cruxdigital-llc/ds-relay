# PR Description Template

When a `/relay-ds:build-component` run completes, the orchestrator produces a PR description summarizing what was built. This is the exact template.

The template has three parts:

1. **Static scaffold** (below) — always the same shape
2. **Auto-populated fields** — filled by the orchestrator from `pipeline-state.yaml` and the various reports
3. **Manual-check checklist** — assembled from the active DS adapter's `manual-checks.md` plus generic items

The output goes to `runs/<run_id>/PR_DESCRIPTION.md` and is the input to any actual PR creation (future `/relay-ds:open-pr` command, or copy-pasted by the user).

---

## Template

```markdown
# <Component Name>

Generated by relay-ds v<version> on <run_started_at>.

## Summary

Component: **<Component Name>** (<n> variants, <n> states)
DS adapter: `<ds_adapter>`
Target repo: `<target-repo-path>`

## Pipeline outcome

| | |
|---|---|
| GIGO score (final) | <score> / 1.00 |
| GIGO penalties applied | <count> — see `reports/gigo-log.yaml` |
| Human gates resolved | <count> |
| Visual Reviewer iterations | <n> / 5 |
| Accessibility remediation rounds | <n> / 3 |
| Quality Gate retries | <n> / 1 |

## What was built

<From architecture.md — a brief summary of:
 - Composition strategy (simple props / compound / discriminated unions)
 - Key dependencies used (with rule references: CR-1, CR-3, AR-2)
 - Any [CONCERN] items that shipped (with rationale)
 - Any deferred features, with pointers to the forward-architecture notes>

## Known tradeoffs (shipped with [CONCERN] rationale)

<For every [CONCERN] marker in any artifact, list:
 - The agent that raised it
 - The message
 - The rationale for proceeding>

## Manual test checklist

Things automation cannot reliably verify. A human must walk through these before merge.

### Generic (pipeline-level)

- [ ] **Screen reader experience** — test with VoiceOver / NVDA / JAWS / TalkBack. Automated a11y tools catch ~30-50% of WCAG violations; the rest requires actual AT navigation.
- [ ] **Hover / mouse-movement paths** — test diagonal cursor movement from trigger to submenu/popover at slow, medium, and fast speeds. Automated `pointerenter` events cannot reproduce movement *paths*.
- [ ] **Motion feel** — confirm durations, easing, and enter/exit sequences feel right in the product context. Wrong motion is worse than no motion.
- [ ] **Visual pixel verification of custom indicators** — 2x zoom side-by-side with Figma reference. Sub-pixel rounding errors in scaled indicators are invisible to automated screenshot comparison.

### Adapter-specific (from `adapters/<ds_adapter>/manual-checks.md`)

<Copy every unchecked `- [ ]` item from the active adapter's manual-checks.md>

### Run-specific ([HUMAN_JUDGMENT] items)

<Copy every [HUMAN_JUDGMENT] marker from visual-review.md, with context>

## Files changed

<Generated from the target repo's diff:
 - <path/to/ComponentName.tsx> — <line count>
 - <path/to/ComponentName.module.css> — <line count>
 - <path/to/ComponentName.stories.tsx> — <line count>
 - <path/to/index.ts> — <line count>>

## Artifacts

- `runs/<run_id>/brief.md`
- `runs/<run_id>/component-rules.md`
- `runs/<run_id>/architecture.md`
- `runs/<run_id>/a11y-report.md`
- `runs/<run_id>/visual-review.md`
- `runs/<run_id>/quality-gate.md`
- `runs/<run_id>/pipeline-state.yaml`

Run `/relay-ds:pipeline-review <run-id>` after merge to feed findings back into the rules ladder.
```

## Populator rules

The orchestrator populates each `<angle-bracket>` placeholder by reading the named artifact:

| Placeholder | Source |
|---|---|
| `<version>` | `.claude-plugin/plugin.json` `version` field |
| `<run_started_at>` | `pipeline-state.yaml` `run_started_at` |
| `<Component Name>` | `pipeline-state.yaml` `component` |
| `<ds_adapter>` | `pipeline-state.yaml` `ds_adapter` |
| `<target-repo-path>` | `pipeline-state.yaml` `target_repo` |
| `<n variants>` / `<n states>` | `brief.md` — count the enumerated variants and states |
| `<score>` | `pipeline-state.yaml` `gigo.score` |
| `<penalty count>` | `pipeline-state.yaml` `gigo.penalties` length |
| `<human gates resolved>` | `pipeline-state.yaml` `gates` — count entries with `status: resolved` |
| Iteration counters | `pipeline-state.yaml` `iterations` |
| `[CONCERN]` items | grep every report for `[CONCERN]` markers, quote with agent name |
| Generic manual checks | fixed list above (per `standards/three-tier-failures.md`) |
| Adapter-specific manual checks | `adapters/<ds_adapter>/manual-checks.md` — copy every top-level `- [ ]` item |
| `[HUMAN_JUDGMENT]` items | `visual-review.md` — copy the `[HUMAN_JUDGMENT]` section |
| Files changed | `git diff --stat` against the target repo, or a filesystem walk of `<target-repo-path>/<output-dir>/` |

## What the PR description is NOT

- **Not a design review.** Design review is upstream — it happened at the conversational architecture gate.
- **Not a code review.** Code review happens after PR creation, by humans reading the diff.
- **Not a comprehensive audit.** Full audit details live in the per-run artifacts; the PR description points at them.
- **Not auto-sent.** v0.1.0 writes the description to a file. The user copies it into their PR tool (GitHub, GitLab, Bitbucket). A future `/relay-ds:open-pr` command will automate this.
